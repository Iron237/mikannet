# Mikannet 系统运行逻辑与判别逻辑

> 本文叙述系统当前(2026-06)的整体运行机制与各处的判别(决策)逻辑,作为理解全局的总览。
> 实现细节随代码演进,以代码为准;此处力求把「为什么这样判、按什么顺序判」讲清楚。

---

## 一、一句话定位与技术栈

**Mikannet** 是一个番剧订阅 / 下载 / 媒体库管理系统:从蜜柑计划(mikanani.me)订阅番剧 → 自动抓种子 → 交给桌面端 qBittorrent 下载到 NAS → 在 WebUI 里管理订阅、控制下载、以「虚拟库」形式精排展示本地番剧库,并按需推送更新。

- **后端**:FastAPI + SQLAlchemy 2.0 + SQLite + APScheduler。
- **前端**:Vue3(Vite / Pinia / Router,零 UI 框架依赖)。
- **下载器**:qBittorrent(主),BitComet(备用后端,能力受限)。两者都直写同一份 NAS(SMB/CIFS)目录。
- **部署**:Docker Compose 单服务 `mikannet`,代码烤进镜像 → 改代码后 `docker compose up -d --build` 重建并重启。
- **数据库**:SQLite,`journal_mode=DELETE`(WAL 在 Windows Docker bind mount 上不稳定),**单进程单写者**——所有写库串行,网络抓取可并发但落库阶段串行。

---

## 二、贯穿全局的几条核心理念

理解下面四条,后面所有逻辑都顺理成章:

### 1. 文件不动 + 虚拟库(ADR-0001)
下载产物**永不移动**(默认),媒体库是数据库里的「虚拟」视图。每个剧集可以挂多个版本文件,库视图只展示其中 `is_active=True` 的那个。v2、画质升级、同集多源去重,统统只是**翻转 `is_active` 标志位**,旧文件保留(仍可做种),不删不动。

> 注:ADR-0002 在此之上加了**可开关的就地整理**——用 qB 的 `renameFile` 把文件原地改名成 Jellyfin 结构(`番剧名/Season SS/番剧名 SxxExx.ext`),只改名不复制,仍按新路径做种。本地导入是唯一会真正移动文件的入口(它处理的不是做种产物)。

### 2. is_active 仲裁:同一集多文件,只留一个最优
当一集对应多个文件(v1+v2、Web+BD、合集里混进的菜单小片……),系统按统一的「画质优先序」选出**唯一**最优者置为 active,其余全部置灰隐藏。优先序见 [§九](#九完成后处理)。

### 3. 容器伪订阅:本地内容也走同一套库视图
本地导入 / 番剧库扫描的文件,不是从 RSS 来的,但为了复用「番剧 → 剧集 → 文件」这套库视图,系统给它们造了**伪订阅**:
- `mikan_subgroup_id = "local"` → 「本地导入」容器订阅;
- `mikan_subgroup_id = "auto"` → 「智能下载」容器订阅。

这两类订阅 `enabled=False`,**不参与 RSS 轮询、不出现在订阅列表**。它们下面挂一个合成的 `ARCHIVED` Torrent 行,本地文件作为 `VideoFile` 挂在其下。

### 4. 元数据失败不阻塞主流程
抓元数据、抓封面、ffprobe 探测、文件整理——这些「锦上添花」的步骤**失败都不回滚主状态**。订阅照建、种子照下、库照展示;失败处留痕、可重试。SMB 抖动、bgm.tv 超时都不会卡住下载链路。

## 三、数据模型速览

| 表 | 角色 | 关键字段 |
|---|---|---|
| **bangumi** | 一季番剧(展示与管理的核心对象) | `mikan_bangumi_id`(可空,唯一)、`bgmtv_subject_id`、`kind`(tv/movie/ova)、`airing_status`(airing/finished)、`auto_best`(是否开智能下载)、`bd_owned`(已购原盘→排除自动下载)、`season_number`、`eps_total` |
| **subscription** | 番剧 + 字幕组 + 过滤规则 | `mikan_subgroup_id`、`include/exclude_keywords`、`pinned/blocked_guids`(手动勾选)、`exclude_batch`、`backfill`(补齐/只追新)、`episode_offset`(跨季编号偏移)、`save_path`、`enabled` |
| **episode** | 番剧的单集 | `number`(支持 12.5,可空)、`type`(regular/special/credits/trailer/other);唯一约束 `(bangumi, type, number)` |
| **torrent** | 一个种子任务(或被淘汰留痕的条目) | `guid`(唯一,= Episode 页 URL)、`info_hash`(唯一,关联 qB)、`status`(状态机)、`is_batch`、`version`、`stalled_since`/`progress_at`(死种与无进度判定计时) |
| **torrent_episode** | 种子 ↔ 剧集 多对多 | 合集种子覆盖多集 |
| **video_file** | 下载产物文件 + ffprobe 信息 | `relative_path`(相对下载根)、`episode_id`(可空=待人工确认)、`source`(Web/BD)、`subgroup`、`resolution`/`color_depth`/`hdr`/音轨/字幕轨、`is_active` |
| **bd_release / bd_extra** | 蓝光发行与特典(ADR-0004) | 与剧集解耦:`source_kind`(bdrip/raw_disc)、`owned`、特典 `category`(sp_anime/pv/gallery/audio…)、`media_kind`(video/audio/image) |
| **notification_config / setting** | 推送通道 / KV 运行时配置 | — |

## 四、番剧如何进入系统(四条入口)

| 入口 | 触发方式 | 匹配数据源 | 产物 | 特点 |
|---|---|---|---|---|
| **蜜柑搜索订阅** | 订阅向导手动建 | Mikan 站内搜索 | 正常订阅(参与 RSS 轮询) | 标准路径 |
| **蜜柑批量导入** | 粘 cookie 导「我的番组」 | Mikan 番剧页 | 仅入番剧库(不自动建 RSS / 不下载) | 支持季度时间范围;可勾「导入后智能下载补齐」 |
| **本地导入** | 扫描指定目录后执行 | bgm.tv 搜索 | 移动文件 + 容器订阅 | 唯一会移动文件的入口 |
| **番剧库扫描** | 扫描下载根目录 | 已有番剧标题 → 否则 Mikan 搜索 | 就地登记(不移动) | 增量同步,与磁盘双向比对 |

### 关键判别:为什么本地内容走 bgm.tv,蜜柑订阅走 Mikan?
**Mikan 的 `/Home/Search` 只索引罗马音 / 拉丁标题**——中文和日文标题搜索返回 0 命中(连 Mikan 自己的中文显示名都搜不到)。而用户给本地文件夹起的名字多为中文。所以:
- 蜜柑订阅 / 批量导入:数据本来就来自 Mikan,用 Mikan 反查;
- 本地导入:文件夹名多为中文 → 一律走 **bgm.tv**(原生索引中文 `name_cn` + 日文 `name` + 罗马音别名);
- 番剧库扫描:先与库里已有番剧做标题匹配(精确 + 归一化),匹配不到才退回 Mikan 容错搜索。

### 本地导入的作品名判别
不靠解析文件名(文件名常是罗马音发布名,匹配不到),而是**用文件夹结构**:取视频的作品文件夹(直接父目录);若父目录是 `Season/Disc/Specials/特典` 等**结构性目录**则再上溯一层。只有散落在扫描根的裸文件才退回 anitopy 解析标题。

### 跨入口去重
- 本地导入 / 库扫描按 **bgm.tv subject** 或 **mikan_bangumi_id** 归一:同一部番剧(哪怕先由蜜柑订阅建过)直接挂上,不建重复。
- 同 `relative_path` 已登记过则跳过,避免产生多行(否则后续 `scalar_one_or_none()` 会抛 `MultipleResultsFound`)。

## 五、元数据判别逻辑(`enrich_bangumi`)

三级降级,任何一级失败都不阻塞:**bgm.tv 全量元数据 → 仅 Mikan 标题/封面 → 手动绑定**。

抓取顺序与取舍:
1. **Mikan 番剧页**(若有 mikan_bangumi_id):补标题、封面缩略图、反查出 bgm.tv subject id。
2. **bgm.tv subject**:中文名 `name_cn` 优先作 `title`,日文 `name` 作 `title_original`;补年份、季度串(`2025冬`)、放送星期、制作公司、简介、评分、总集数。bgm.tv 封面质量优于 Mikan 缩略图 → 覆盖。
3. **TMDB 背景图**(需配 key):多候选查询——原名 → 中文名 → **去掉季/期后缀**(TMDB 按「剧集」建条目,不带季号),命中即缓存横版背景。

**番剧形态(`kind`)判别**:
- 若已绑 AniDB(ADR-0003,默认关)→ 由 AniDB anime type 主导(更可靠);
- 否则由 bgm.tv `platform` 推断:含「剧场版/劇場版」或 `movie` → **MOVIE**;`OVA/OAD/OAV` → **OVA**;其余(TV/WEB…)→ **TV**。
- `kind` 至关重要:**非 TV(电影/OVA)的文件不归正片集**,避免剧场版被错当成「第 1 话」。

**放送状态(`airing_status`)判别**:无官方字段,以「放送开始 + 集数」粗推——开播超过 `集数 + 4 周` 视为**完结**,否则**连载中**。它决定补番时合集的默认策略(连载默认排除合集,完结优先合集)。

**续作季号(`season_number`)判别**:从标题猜——`第二季 / 2nd Season / Season 3 / SⅡ / Ⅲ`,猜不到为 1。用于 Jellyfin 的 `Season N` 目录与整理改名。

## 六、标题解析判别逻辑(`title_parser` —— 全系统的判别基石)

标题解析把一条原始种子标题 / 文件名,解析成:`集数 / 版本(v2) / 是否合集 / 分辨率 / 字幕组 / 片源 / 剧集类型 / 置信度`。**去重、v2 判定、虚拟库映射、智能下载、推送文案全都依赖它**。解析失败不抛错,返回低置信度,由调用方决定「下载但待人工确认」。

策略链(依据 P0 阶段 358 条真实语料的失败模式设计),**按顺序命中即返回**:

0. **预处理**:全角转半角、`★`(星号分隔格式)转空格、去零宽字符、压缩空白。
1. **合集判定**:命中合集关键词(`整理搬运|合集|全集|Batch|BOX|Fin`)或集数范围(`[01-12]`、`[25-48END]`、`【01-24】`)→ 标记 `is_batch`,展开范围为集列表。范围有「日期形」(`2016.12.31`)保护,不误判。
2. **PART / 子集**:哆啦A梦式 `[164PART1]`、`[879-2]`、`[467-B]` → 取主集号(子集粒度不建模)。
3. **方括号内独立集号**:`[02]/[04]/[123]`(括号内只有 1–3 位裸数字,排除分辨率裸数)→ 优先采信。这是为了对付「续作季用 `标题 N [集]` 命名」时,anitopy 把裸季号 N 误当集号的情况——方括号里的裸数字才是真集号。
4. **anitopy 主解析**:拿集号、版本号;若解析成两个相距 >1 的端点 → 视为合集范围。
5. **中文集数兜底**:`第08话 / 第8.5集 / [08话]`。
6. **整季打包兜底**:有 `S01 / S01v2` 季标记但无单集号 → 合集。

**剧集类型(`ep_type`)判别**(按特异性从高到低,命中即返回):
- **credits**:`NCOP/NCED/OP12/ED1/片头/片尾/Creditless`(最明确,最先判);
- **trailer**:`PV/CM/Trailer/预告`;
- **special**:强标记 `SP/OVA/OAD/剧场版/映画/Eiga/电影/Movie/Special`,或弱描述 `特别篇/特典/番外/总集篇`;
- **other**:`Menu/Logo/Bonus/Sample/花絮/MAD/访谈`;
- 都不命中 → **regular**(正片)。

**两条解析后修正(压住误判)**:
1. 仅**弱特典描述**(特别篇/特典/番外…)且带明确正片集号时,**集号说了算 → 让位给正片**(如哆啦A梦「[109]…特别篇…」仍是正片第 109 话);**强标记**(SP01/OVA/剧场版/OP·ED/PV)不让位。注意只降级 `special`,**绝不碰 `other`**——否则 BD 合集里的 `[menu][01]` 会被当成正片第 1 话,与真第 1 话撞名。
2. 非正片里抽到的「年份」不是集号(`剧场版 2023` ≠ EP2023)→ 丢弃。

**LLM 兜底**:置信度 < 0.5 时,若设置里开了 LLM,调用一次大模型补解析(失败不影响规则结果)。

**片源(`source`)判别**:`BDRip/Blu-Ray/REMUX/UHD-BD` → **BD**(更具体,先判);`WEB-DL/Baha/Bilibili/CR/AMZN/Netflix/爱奇艺…` → **Web**;判不出 → None。`detect_source()` 公开版还能对文件夹名 / 路径段判片源(库扫描用文件夹上下文补片源)。

**字幕语言判别**(`detect_subtitle_tags`,搜索页 chips 与智能下载用):按特异性取**最高优先级的一个**语言标签(`简繁日 > 简日 > 繁日 > 简繁 > 简体 > 繁体`,避免「简日+简体」冗余)+ 若干封装标签(内嵌/内封/外挂)。

## 七、RSS 订阅下载主管线(状态机)

`rss_engine` 是**唯一向 torrent 表写入新行的入口**。RSS 轮询与「补齐历史」共用这一条管线(数据源同为 RSS,全量历史)。

### 单条 RSS 条目走过的判别链(`process_item`)

```
新条目
  └─ guid 已存在? ── 是 → 直接忽略(去重第一道:guid 唯一)
       否
       └─ 标题解析
          └─ 手动偏差优先于规则:
               · guid ∈ blocked_guids → 强制排除(SKIPPED「手动排除」)
               · guid ∈ pinned_guids  → 强制通过(跳过关键词与去重)
               · 否则走过滤规则
          └─ 过滤(passes_filters):
               · include 关键词:AND 语义,缺任一即排除
               · exclude 关键词:任一命中即排除
               · is_batch 且订阅设了 exclude_batch → 排除
               不通过 → SKIPPED(留痕,记原因)
          └─ 去重 + v2 判定(_dedup_verdict):
               · 集数未知(低置信度)→ 不拦,下载后人工确认
               · 同订阅、状态非 SKIPPED/SUBMIT_FAILED 的种子里,
                 找「集合覆盖本条目集数」的既有种子
                 - 无 → 接受
                 - 有,且本条目 version 严格更高 → 接受(v2 路径)
                 - 否则 → SKIPPED(重复)
          通过 → PENDING [emit on_new]
          └─ 自动集数偏移检测(_auto_detect_offset):
               集号 > 总集数(如二季给的是 25-48 / 本季 24 集)
               → 推断偏移 = 24,后续集号减偏移得本季集号(可手动改写)
          └─ 建 episode 行 + torrent_episode 关联
          └─ 提交下载器(_submit,最多重试 3 次):
               · 下载 .torrent 字节 → qB add(带 category + save_path)
               成功 → DOWNLOADING [emit on_start],记 info_hash
               重试耗尽 → SUBMIT_FAILED [emit on_fail]
```

### 「只追新」与「补齐」的判别
- `backfill=True`(补齐):看全部历史条目。
- `backfill=False`(只追新)且是**首轮**:不入库任何 guid(入库会污染留痕),只记录当前最新发布时间为基线,全部跳过;之后只处理比基线新的条目。

### 留痕条目的「复活」(`reevaluate_skipped`)
过滤规则改了、或坏种换源时,重新评估本订阅里 `SKIPPED` 的条目:符合条件的重新走管线提交。但**「手动删除」和「坏种自动清理」标记的不复活**(否则会循环重下死种)。

### 轮询调度(`poll_all`)
- 网络阶段并发(8 线程拉 RSS),落库阶段串行(SQLite 单写者)。
- **排除**:`local`/`auto` 容器伪订阅(无单一 RSS 源)、已购原盘的番剧(`bd_owned`,ADR-0004 整体跳过)。
- 单订阅失败不影响其他,记 RSS 健康状态(`last_poll_ok` / `last_poll_error`)。

## 八、下载跟踪:状态迁移 + 坏种 + 无进度暂停(`download_tracker`)

后台异步轮询 qB,回写进度快照并广播 WebSocket。**自适应频率**:有活动任务 2s 一轮,空闲 10s 一轮。

### 状态迁移判别(每轮对每个 DOWNLOADING/COMPLETED/DOWNLOAD_ERROR 任务)

```
DOWNLOADING:
  · qB 里找不到这个 hash → DOWNLOAD_ERROR(可能被手动删除)
  · qB 报 error → DOWNLOAD_ERROR
  · qB 已完成(done / progress≥1) → COMPLETED [on_complete],入队后处理
  · 否则依次过两道「卡住」检测(见下),都没触发 → 计为「活动中」

DOWNLOAD_ERROR(自愈):
  · qB 侧已完成 → COMPLETED,补走完成迁移
  · qB 侧恢复成活跃下载态(downloading/stalledDL/metaDL/forcedDL/queuedDL/
    checkingDL/allocating)→ 回 DOWNLOADING,重置计时
  · 注意:pausedDL/stoppedDL 不在恢复名单 → 无进度被暂停的任务保持暂停,不被打回
```

### 两道「卡住」检测(可独立开关,并存)
1. **无进度暂停**(`stall_pause`,温和档,默认 12h):只要进度还在涨就持续刷新计时;停涨超过阈值 → **暂停**(不删),标 `DOWNLOAD_ERROR` 复用「错误→恢复」入口,手动恢复时重置计时。
2. **坏种自动清理**(`dead_torrent`,默认 6h):**0 做种 + 0 速** 持续超过阈值 → **删除任务+文件** → 标 `SKIPPED「坏种自动清理」`(不复活)→ 自动 `reevaluate_skipped` 换一个备选源。

> 设计取舍:无进度暂停先于坏种删除触发的可能性取决于阈值(12h vs 6h);坏种针对「彻底没人做种」,无进度针对「有种但就是不动」。两者计时字段独立(`stalled_since` / `progress_at`),恢复时都会清零,避免残留计时导致刚恢复就被删。

### 启动对账
重启后:DB 里 DOWNLOADING 但 qB 没有 → DOWNLOAD_ERROR;qB 已完成 → 补走完成迁移并入队后处理。

## 九、完成后处理(`postprocess`)

串行 worker 线程消费队列(**SMB 上禁止并发 ffprobe**)。探测失败停在 COMPLETED 留错误标记可重试,不阻塞通知与库展示。

### 流程
1. 列出种子的所有文件,逐个视频文件登记 `VideoFile`。
2. **文件 → 剧集映射(`_match_episode`)**:
   - 番剧 `kind != TV`(电影/OVA)→ **不归正片集**,文件留作「影片本体」;
   - 单集种子(非合集、只关联一集)→ 直接用关联集;
   - 合集 → 按**文件名**重新解析定集,按 `ep_type` 归类:**正片**减集数偏移后入正片集;**非正片**用自身小序号(SP1/OVA03)或留空,不减偏移、不占正片集号。
3. **ffprobe 探测**:分辨率、视频编码、色深(从 pix_fmt 推 8bit/10bit)、HDR(HDR10/HLG/DV)、码率、逐条音轨/字幕轨。
4. **is_active 仲裁(`_apply_version_switch`)**:同一集的所有文件按画质优先序选**唯一**最优为 active:

   > **画质优先序(越小越优)**:`片源(BD=0 > Web=1 > 未知=2)` → `版本号高` → `体积大` → `id 稳定`。
   > 体积兜底很关键:同集挂了多个 BD 文件(真正片 + 被误映射的菜单/NC 小片)时,最大的才是正片。

   这一步统一覆盖了:v2 决议、跨字幕组 Web→BD 升级、同集多源去重(「同一集多个源只留一个」)。

### 落库与整理的顺序(防 qB 抖动)
**先把文件映射 + `ARCHIVED` 落库并 commit**,再做整理(qB 改名)。即便整理出错也不回滚文件记录、不让种子卡回 COMPLETED——否则会被 `drain` 反复重入队 → 反复改名 → 搅乱 qB。

### 就地整理(`organize`,ADR-0002)
仅 qB 后端、`organize_enabled` 开启时:把 active 且已定集的**正片 + 带集号的特别篇**用 `renameFile` 原地改名成 `Season SS/番剧名 SxxExx.ext`,根目录写 `tvshow.nfo` + `poster.jpg` + `fanart.jpg`。菜单/OP·ED/PV/无集号特典**留原名**(Jellyfin 当 extras)。预判撞名跳过,绝不让 flush 报错毒化事务。

## 十、智能下载判别逻辑(`auto_best`)

目标:扫一部番剧的所有字幕组种子,按偏好挑最佳源——**补全**库里没有的正片集,**升级**已有 Web 为合格 BD。种子挂到 `auto` 容器伪订阅,复用下载/后处理管线。

### 硬过滤(候选准入,`_candidate`)
缺一不可,否则该种子直接淘汰:
- 必须能解析出集数或是合集;
- **分辨率严格等于**目标(默认 `1080p`);
- **必须含偏好字幕语言**(默认简中:标题须含简体类标签)。

### 何为「需要处理」(`needed`)
对每个候选集:
- 库里**没有**这一集 → 补全(若开 `do_fill`);
- 库里**有 Web/未知**,且出现**合格 BD** → 升级(仅「→BD」升级;不靠「Web>未知」这类弱信号升级,避免把没标 BD 标记的蓝光误降级)。
- 已购原盘(`bd_owned`)的番剧整体跳过。

### 选种:同字幕组优先 + 去重(最新逻辑)
> **背景**:旧贪心逐集挑最优,会出现「逐集源覆盖 03–11、再拉一个别家整包合集补 01–02 缺口」→ 03–11 被重复下载,而那种补缺合集常是死种。用户要求「优先匹配同一个字幕组的源 + 不重复添加」。

现在采用**子组锚定式选种**:

```
covered ← 已在库且无需升级的集(防跨次重复下载)
remaining ← 需要处理的集
while 还有缺集:
  1) 在「能覆盖至少一个仍缺集」的字幕组里,挑最优子组:
     片源(BD>Web) → 画质(子组里最高分) → 可用性(发布越新种子越活)
     → 对仍缺集的覆盖数(同字幕组尽量整部覆盖)
  2) 在该子组内,按 画质→合集优先→体积 排序,逐个取种子;对每个:
     · 不带来任何新缺集 → 跳过
     · 重叠「已认领/已在库」的集 数量 > 它新带来的集 → 判为浪费,跳过
       (典型:为补 1–2 集去下整包 01–11)
     · 否则选中,把它的整段集范围计入 covered(后续不再重复下)
  3) 处理完该子组即移出候选,看下一个
```

**判别要点**:
- **片源 / 画质 / 可用性是子组排序的前置维度**,所以死种、低画质的整包合集不会因「覆盖集多」而被优先(可用性用「发布时间越新」作种子越活的代理,因蜜柑不给做种数)。
- **「重叠多于新增即跳过」**是去重的核心:既杜绝了「逐集源 + 别家整包」的重复下载,也顺带躲开了为小缺口下大死种合集。代价是:此时缺口(如 01–02)留待下次有更贴合的源(逐集或小合集)再补,可能暂时缺集。
- 零重叠的跨字幕组拼接是允许的(如 A 组覆盖 1–6、B 组覆盖 7–11)。
- 同集多源(BD vs Web)只会留 BD —— 片源维度直接淘汰 Web。

### 画质评分(`_quality_score`,满足硬过滤后的软排序)
`10bit/Hi10P` +4、`HEVC/x265` +2(`AVC/x264` +1)、`FLAC/无损` +1。同分再比体积(码率)。

### 入口
- 番剧库页勾选 / 详情页按钮 → 手动扫(`start_scan`);
- 定期任务 `_auto_best_job` → 扫所有开了 `auto_best` 的番剧(默认 360 分钟,可设 0 关闭);
- 蜜柑批量导入勾「导入后智能下载」→ 对本次新建番剧触发一次。
- 前端有**进度列表**:展示「进入下载任务前」每部已选源(来源/画质/集范围/标题)、候选与缺口计数、当前扫描项。

## 十一、BD 特典管理(`bd_scan`,ADR-0004)

把 BD 当「发行实体 + 特典」管理,**与剧集/VideoFile 解耦**(正片仍走剧集网格)。

两类来源:
- **下载根里片源=BD 的合集**(BDRip)→ `bd_release(bdrip, owned=false)` + 逐项分类特典;
- **已购原盘目录** `/bd-owned`(MAKEMKV 原盘)→ `bd_release(raw_disc, owned=true)`,v1 仅占位(碟数 + 总大小,不拆 m2ts),并把对应番剧标 `bd_owned=True`(整体排除自动下载)。

**特典分类判别(`_classify`)**:先看所在子目录名(`特别动画→sp_anime`、`menu`、`NCOP/NCED→credits`、`PV/预告→pv`、`图集→gallery`、`Scans/书子→scans`、`CD/原声→audio`),再看文件名标记;顶层视频若是单话正片则**跳过**(交剧集网格),其余归 `other`,散落图片归 `scans`。

**番剧绑定**:仅匹配「已存在」的番剧(按净化标题),匹配不到留空待手动绑——**不自动 Mikan 建,避免错绑**。

> 库扫描会跳过裸盘结构(`BDMV/STREAM/VIDEO_TS`)与特典子目录,避免把特典/m2ts 当剧集逐个登记。

## 十二、定时任务总览(APScheduler)

| 任务 | 默认间隔 | 作用 |
|---|---|---|
| `rss_poll` | 15 分钟 | 并发轮询所有启用订阅,跑主管线 |
| `drain_completed` | 5 分钟 | 把卡在 COMPLETED 的种子重新入队后处理(幂等,防重启/积压漏掉) |
| `auto_best` | 360 分钟(可关) | 扫所有开了智能下载的番剧 |

另有进程内的**后处理 worker**(队列消费)与**下载跟踪 loop**(自适应 2s/10s),由 FastAPI lifespan 启停;启动时各做一次对账。

## 十三、全局判别规则速查表

| 判别点 | 规则 |
|---|---|
| **能否搜到番剧** | Mikan 只认罗马音;中文/日文 → 走 bgm.tv |
| **条目去重(第一道)** | `guid` 唯一,已存在直接忽略 |
| **过滤** | include=AND、exclude=任一命中、合集看订阅 `exclude_batch` |
| **手动偏差** | `blocked_guids` 强制排除 > `pinned_guids` 强制通过 > 关键词规则 |
| **同集去重 / v2** | 既有种子覆盖本集且版本不低于本条 → 跳过;本条版本严格更高 → 接受 |
| **集数偏移** | 集号 > 总集数 → 自动推断偏移(跨季连续编号→本季集号) |
| **提交重试** | 3 次,耗尽 → SUBMIT_FAILED |
| **坏种** | 0 做种+0 速 > 6h → 删+换源(不复活) |
| **无进度暂停** | 进度停涨 > 12h → 暂停(可手动恢复) |
| **错误自愈** | qB 侧恢复成活跃下载态 → 回 DOWNLOADING;暂停态不打回 |
| **文件归集** | `kind != TV` 不归正片;非正片不占正片集号;弱特典带集号让位正片 |
| **同集多文件 active** | 片源(BD>Web>未知)→ 版本 → 体积 → id,取唯一最优 |
| **智能下载准入** | 分辨率严格相等 + 含偏好字幕语言 |
| **智能下载选种** | 子组锚定:片源→画质→可用性→覆盖数;重叠多于新增即跳过(去重) |
| **整理改名** | 仅 qB、仅正片+带集号特别篇;菜单/OP·ED/PV 留原名 |
| **BD 绑定** | 仅匹配已存在番剧,不自动建 |
| **轮询排除** | local/auto 伪订阅、已购原盘番剧 |
